Philosophy

なぜ「評価基盤」が必要なのか

大規模言語モデル（LLM）の性能評価は、いまや研究の基本動作になりつつある。 しかし「スコアが出た」とき、私たちは本当にそれを信じてよいのだろうか。

結果だけが残り、過程が消える評価に、科学としての価値はない。
私たちが追うべきは最高スコアではなく、同じ条件で、誰でも、もう一度再現できることだ。

GPU の世代、ドライバのバージョン、推論バックエンド、生成パラメータ—— 日本語LLM評価はこれらの「見えない変数」の影響を受けやすい。 スコアの数字だけを比べる行為は、異なる実験条件を同列に並べる危険を常にはらんでいる。

MAGI は、この問題に対して仕組みで答える。 評価の「実行環境・仕様・結果・失敗」を不可分な成果物（Artifact）として一体保存し、 後から誰が見ても追跡・再現できる基盤を構築する。

Design Principles

4つの設計原則

01

Spec is Canonical

実験の定義は spec.json で一意に定まる。 同一 spec は同一条件での再実行を保証する。 曖昧な「設定ファイル」ではなく、実験の契約書として機能する。

02

Artifacts-first

成功も失敗も、成果物（Artifact）として保存する。 主張はすべて Artifact にリンク可能でなければならない。 「動いた」ではなく「証拠がある」が基準。

03

Never Lose Evidence

失敗は資産である。ビルドエラー、OOM、数値的不整合—— すべてを分類・蓄積し、再現性の改善に活かす。 削除より隔離。破壊せずに正す。

04

Plane Separation

制御（Control）と計算（Compute）を物理的に分離する。 推論エンジンの依存がパイプラインを壊さない。 異なる GPU・異なる環境を、同じ契約で束ねる。

Architecture

3-Plane アーキテクチャ

実験の定義・実行・保存を分離し、ハードウェア差を吸収する設計。 Control Plane は推論を一切行わず、Compute Plane との依存を断つ。

Control Plane

DL325 Gen10 / EPYC

  • MCP Server (TypeScript)
  • Runner CLI (Python)
  • MAGI Agents
  • 実験定義・統合・監査
Compute Plane

ROCm / CUDA Nodes

  • Radeon R9 7900 XTX AI Pro
  • GeForce RTX 3060
  • llama.cpp / vLLM
  • 推論実行・env.json 生成
Knowledge Plane

Tank (ZFS 5.5TB)

  • artifacts/runs/{run_id}/
  • datasets/ & models/
  • docs-ref/ (参照資料)
  • audit_logs/ (監査記録)

Control → Compute: SSH + 固定コマンド　/　Compute → Tank: Artifact 書き戻し　/　Control ← Tank: 統合・分析

Artifact Structure

統合成果物 — result.json

1つの実験実行（run）が生成するすべての証拠を、単一の統合ファイルに集約する。 仕様・環境・応答・メトリクス・整合性ハッシュが不可分に結合される。

tank/artifacts/runs/{run_id}/ ├── spec.json # 実験仕様（契約） ├── env.json # 実行環境の証拠 ├── responses.jsonl # 生の推論出力（raw evidence） ├── result.json # 統合成果物 │ ├─ schema_version │ ├─ run_id, created_at │ ├─ spec + spec_hash │ ├─ env (GPU / OS / driver) │ ├─ responses[] + gen_params │ ├─ metrics │ └─ integrity (SHA-256) ├── report.md # 人間向け分析レポート └── logs/ # 実行ログ

integrity.responses_sha256 と integrity.env_sha256 により、 成果物が事後に改変されていないことを暗号学的に検証できる。

Evaluation Pipeline

評価パイプライン

Step 1

実験定義 — submit_experiment

spec.json を登録し、一意な run_id（YYYYMMDD-HHMMSS-{hash}）を生成。MCP Server が管理。

Step 2

推論実行 — Compute Node

spec に基づき Compute Node で推論を実行。responses.jsonl と env.json を生成。

Step 3

統合・検証 — runner integrate

JSON妥当性チェック → メトリクス算出 → ハッシュ計算 → result.json を原子的に書き込み。

Step 4

分析・監査 — MAGI Agents

Result Analyst が自動分析。Reviewer がコードを査読。すべて監査ログに記録。

Contribution

本研究の主張

LLM評価の信頼性は、モデルの能力ではなく、評価手順の透明性に依存する。
本研究は、日本語LLM評価において実行環境・仕様・結果を統一的に記録し、
再現可能な検証パイプラインを実装・実証した。
  1. 再現可能な評価パイプラインを実装し、定義から統合まで1周の実行を実証した
  2. 実行環境情報（GPU・OS・ドライバ）と評価結果を統一フォーマットで一体保存できる
  3. Control / Compute Plane の分離により、ハードウェア差を吸収する運用設計を示した
  4. 失敗を含むすべての実行記録を Artifact として保全する証拠保全の仕組みを構築した
